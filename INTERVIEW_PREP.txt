--------------------------------------------------------------------------------
1. EXECUTIVE SUMMARY
--------------------------------------------------------------------------------
Voyager-Agents is a distributed Multi-Agent System (MAS) built to solve the 
"Complex Travel Planning" problem. It utilizes a Directed Acyclic Graph (DAG) 
managed by LangGraph to orchestrate 7 specialized agents. The system transforms 
unstructured user intent into a high-fidelity, data-rich travel package 
including logistics, cultural insights, and visual assets.

--------------------------------------------------------------------------------
2. ARCHITECTURAL PILLARS
--------------------------------------------------------------------------------
A. ORCHESTRATION LAYER (LangGraph):
- STATEFULNESS: Uses a central `TripState` (TypedDict) that persists data 
  across node transitions. This prevents data loss and allows agents to 
  contextualize their tasks based on previous findings.
- NODES: Each node is an asynchronous Python function representing an agent.
- EDGES: Control the flow. We use "Conditional Edges" to decide whether to 
  continue planning or stop if an error occurs.
- PARALLELISM: The graph is designed to branch after the Research phase. 
  Flights, Hotels, and Weather agents execute in parallel, leveraging 
  Python's `asyncio` for maximum throughput.

B. INTELLIGENCE LAYER (Google Gemini 2.5 Flash):
- REASONING: Gemini acts as the "Brain" for each node, interpreting search 
  results and structuring them into JSON.
- SPEED: The "Flash" model was chosen specifically to keep the end-to-end 
  latency under 45 seconds despite making 10+ LLM calls.

C. DATA RETRIEVAL LAYER (Tavily & Wikipedia):
- RAG (Retrieval-Augmented Generation): Instead of relying on internal 
  knowledge, the system performs real-time retrieval.
- TAVILY: Used for "LLM-optimized" search results. It returns clean content 
  without HTML clutter, which reduces token usage and improves LLM accuracy.
- WIKIPEDIA: Used for canonical cultural and historical data to ensure 
  educational value in the itinerary.

--------------------------------------------------------------------------------
3. THE MULTI-AGENT COMPOSITION (DEEP DIVE)
--------------------------------------------------------------------------------

AGENT 1: RESEARCH AGENT (The Foundation)
- Input: Destination Name.
- Prompt Strategy: "Act as a cultural historian and travel scout."
- Logic: Fetches the 'Hero Image' URL and general city background. 
- Why First?: It validates if the destination is real and provides the 
  context (e.g., currency, language) that other agents need.

AGENT 2: PLACES AGENT (The Discovery Engine)
- Logic: Searches for "top tourist attractions."
- Image Handling: For every place found, it makes a sub-call to fetch a 
  specific image URL, ensuring the UI is visually immersive.

AGENT 3: FLIGHTS AGENT (The Logistics Specialist)
- Logic: Simulates real-time pricing and airline availability.
- Currency: Handles conversion (e.g., providing prices in INR for Indian users).

AGENT 4: HOTELS AGENT (The Accommodation Expert)
- Budget Awareness: Uses a "Budget Filter" logic. If the user selects 'Low', 
  it prioritizes guest houses/hostels. If 'High', it looks for 5-star resorts.

AGENT 5: WEATHER AGENT (The Safety Officer)
- Critical Role: If the weather is dangerous (e.g., Monsoon in India), it 
  adds warnings to the itinerary.

AGENT 6: ACTIVITIES AGENT (The Personalizer)
- Interest-Driven: It doesn't just suggest popular things; it maps the user's 
  specific interests (Food, Adventure, History) to local events/experiences.

AGENT 7: ITINERARY AGENT (The Master Weaver)
- Synthesis: This is the most complex prompt. It takes ~50KB of raw JSON 
  data from all previous agents and distills it into a human-friendly Markdown.
- Formatting: Uses Markdown tables, bold headers, and bullet points.

--------------------------------------------------------------------------------
4. THE BACKEND SYSTEM DESIGN (Python/FastAPI)
--------------------------------------------------------------------------------
- ASYNC/AWAIT: The entire backend is non-blocking. This is crucial because 
  waiting for 7 LLM responses sequentially would take minutes.
- ERROR HANDLING: Every agent call is wrapped in a Try-Except block. If one 
  agent fails (e.g., Weather API is down), the graph doesn't crash; it 
  records the error and allows the Itinerary Agent to apologize to the user.
- PYDANTIC MODELS: We use strict schemas for the `TripFormData` and 
  `TripPlanResponse`. This ensures the Frontend never receives malformed data.

--------------------------------------------------------------------------------
5. THE FRONTEND SYSTEM DESIGN (Next.js 14)
--------------------------------------------------------------------------------
- SERVER-SIDE RENDERING (SSR): Used for the initial page load for SEO.
- CLIENT-SIDE STATE: Uses React `useState` to manage the complex loading 
  sequence.
- GLASSMORPHISM UI: 
  - `bg-slate-950/50 backdrop-blur-md` for headers.
  - `border-slate-800` for subtle, premium separation.
  - `animate-fade-in` and `animate-slide-up` for smooth transitions.
- DYNAMIC IMAGE RENDERING: Handles fallbacks gracefully. If an image URL is 
  broken, it displays a themed gradient placeholder.

--------------------------------------------------------------------------------
6. DATA FLOW: THE "LIFECYCLE OF A TRIP"
--------------------------------------------------------------------------------
1. CLIENT SUBMIT: Form data is validated on the client side.
2. BACKEND ENTRY: `app.main.py` receives the POST request.
3. GRAPH EXECUTION:
   - Start Node -> `research_agent`
   - Parallel Nodes -> `flights`, `hotels`, `weather` (runs concurrently)
   - Sync Point -> LangGraph waits for all 3 to return.
   - Sequential Nodes -> `activities` -> `itinerary`
4. RESPONSE: The final `TripState` dictionary is converted to a JSON object 
   and sent to the client.

--------------------------------------------------------------------------------
7. INTERVIEW QUESTIONS & MODEL ANSWERS
--------------------------------------------------------------------------------

Q: Why use LangGraph instead of a simple Sequential Chain?
A: "Sequential chains are brittle. Travel planning is non-linear. LangGraph 
allows for cycles, parallel branching, and complex state management, which 
is necessary for a robust Multi-Agent System."

Q: How do you handle LLM Hallucinations in the itinerary?
A: "We use Retrieval-Augmented Generation (RAG). By forcing the LLM to only 
use the search results provided in the context window from Tavily and 
Wikipedia, we minimize its reliance on internal 'stale' knowledge."

Q: How did you optimize for performance?
A: "Parallel node execution and the use of a 'Flash' model (Gemini 2.5 Flash). 
We also optimized the search queries to be as specific as possible to 
reduce noise in the context window."
